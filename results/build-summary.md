# Build-Artifact Spike — Preliminary Results

**Date:** 2026-06-25
**Host:** Mac Studio (Apple Silicon), macOS Darwin 25.5.0, APFS
**Status:** Both TS and Rust scenarios complete.
**Raw data:** [`results/build/`](build/)

This is the second of the two pre-Phase-0 spikes, per [`ivk_build_artifact_spike.md`](../ivk_build_artifact_spike.md). It validates whether the cheap-workspace claim survives once each workspace actually runs a real build.

---

## TL;DR

For a small but real Vite + React + TS project (with `node_modules` and `dist` baked into the fixture so the comparison reflects "naive duplication" rather than pnpm's already-shared store):

### N = 10 parallel workspaces

| stage | A (git worktree) | **G (ivk clonewt)** | savings |
|---|---:|---:|---:|
| post-clone disk | 705 MB | **16 MB** | **44× less** |
| post-rebuild disk | 704 MB | **34 MB** | **21× less** |
| create time | 6.2 s | **1.8 s** | 3.4× faster |
| rebuild time (10 × `pnpm build`) | 27.3 s | 27.9 s | parity |

### N = 50 parallel workspaces

| stage | A (git worktree) | **G (ivk clonewt)** | savings |
|---|---:|---:|---:|
| post-clone disk | 3.43 GB | **93 MB** | **37× less** |
| post-rebuild disk | 4.01 GB | **500 MB** | **8× less** |
| create time | 31.8 s | **8.7 s** | 3.6× faster |
| rebuild time (50 × `pnpm build`) | 147 s | 147 s | parity |

**The cheap-workspace pitch survives** real builds, but with the understood caveat: each workspace pays its own fresh `dist/` cost (~8-12 MB in our fixture). Working-tree + node_modules sharing wins big (37×); per-workspace build outputs add a linear floor that no amount of clonefile magic can erase.

Working hypothesis confirmed: **ivk + per-workspace builds is still 8-21× cheaper than git worktree at N=10-50, scaling linearly.**

### Rust / Cargo — even more dramatic

For a small Rust binary with realistic deps (serde, tokio, clap; cold `target/` is ~80 MB) at **N = 5 parallel workspaces**:

| approach | post-clone disk | post-rebuild disk | rebuild time |
|---|---:|---:|---:|
| A (git worktree) | 410 MB | **1.25 GB** | **173 s** (each ws does a full rebuild from scratch) |
| **G (ivk clonewt)** | **1.7 MB** | **1.8 MB** | **0.85 s** (cargo sees cloned target/ as valid, does nothing) |
| **G + shared CARGO_TARGET_DIR** | 2.3 MB | 200 MB | 41 s (first ws builds, rest reuse the shared target) |

For Rust, **ivk default already gives a 700× disk reduction and a 200× build-time reduction** on the no-source-change case. The shared-target variant is what to recommend for agents that *modify* source code: one shared 200 MB target/ vs A's 5 × 250 MB = 1.25 GB.

This is the strongest cell in either scenario. The reason it works so well: APFS clonefile preserves cargo's `target/.fingerprint/` files exactly, so cargo's incremental build cache is inherited by every new workspace.

## Methodology

### Fixture

A handwritten Vite + React + TS scaffold under [`.cache/fixture-vite/`](../.cache/fixture-vite/) generated by [`scripts/bench/build/setup-vite.sh`](../scripts/bench/build/setup-vite.sh). Includes 80 generated component files to give the bundler non-trivial work. After `pnpm install` + `pnpm build`, the fixture has:

- `src/`         ~250 KB  (80 small TS files)
- `node_modules/` 64 MB   (real packages: react, vite, typescript, @types/*)
- `dist/`        ~150 KB  (Vite production bundle)

All three are committed to the fixture's git repo so approach A (worktree) materializes them in every worktree — modeling the "naive duplication" baseline.

### Scenarios run

- **A** — `git worktree add`. Each worktree checks out the full fixture including `node_modules` and `dist`.
- **G** — `clonewt` (the Rust prototype from the working-tree spike, now ported to `crates/ivk-core`). Working tree is APFS-cloned from the fixture; `.git/` is shared via worktree pointer.

After cloning N workspaces, each runs `pnpm build` (serial). Both stages are measured for disk delta via `df` free-space sampling.

### Rust fixture

A handwritten Cargo binary under [`.cache/fixture-rust/`](../.cache/fixture-rust/) generated by [`scripts/bench/build/setup-rust.sh`](../scripts/bench/build/setup-rust.sh) with the dep set: `serde`, `serde_json`, `clap`, `anyhow`, `thiserror`, `tokio` (multi-thread). Plus ~10 small generated modules to give compile time some volume. Initial `cargo build --release` takes ~45 s and populates `target/` to ~80 MB. Everything (including `target/`) is committed to the fixture's git so approach A materializes it on each worktree.

### Rust scenarios run

- **A** — `git worktree add` from the fixture.
- **G** — `clonewt`; the working tree (including `target/`) is APFS-cloned.
- **G-shared** — `clonewt` + `CARGO_TARGET_DIR=$ROOT/.cache/cargo-target-shared/` so every ws compiles into one shared directory.

After cloning N workspaces, each runs `cargo build --release` serially. Disk delta measured via `df`.

### Scenarios NOT run

- T4 Turbo remote-cache, R3 sccache: tooling not installed. Out of scope for the spike.
- D2/D3 distinction with per-workspace `pnpm install`: pnpm's content-addressed store already provides T3-equivalent disk sharing by default; the bigger gap we care about is **flat checkout vs clonefile**, which the A-vs-G comparison covers directly.
- Rust with source modifications: our N=5 run measured the "clone and rebuild without source changes" path. A future spike should also measure "clone, modify one file, rebuild" to validate that cargo's incremental rebuild correctly invalidates clonefile-shared state. Expected behavior: incremental rebuild touches only the modified crate, leaving the rest of `target/` block-shared.

## Why the numbers look the way they do

### Why does `dist/` add disk to both A and G post-rebuild?

For A: each worktree starts with the committed `dist/`. The rebuild overwrites those same files. Net delta ≈ 0 — but the build produces *slightly* different output each time (Vite's chunk-hashing or tsbuildinfo timestamps), so per-workspace ~11 MB of fresh disk is consumed.

For G: each workspace starts with a *cloned* `dist/`. When `pnpm build` writes new files, clonefile sharing breaks for those files — they become independent copies on disk. Per-workspace cost: ~8 MB.

The two numbers are similar because the *fresh write* dominates either way. ivk doesn't magically share build outputs across workspaces — that's the next product layer (build cache).

### Where ivk's lead actually comes from

**`node_modules`** (~64 MB per worktree for A vs ~0 incremental for G). At N=50, this dominates: A has 50× the node_modules + dist that ivk has 1×.

### Why is build time the same?

Bundling is CPU-bound, not disk-bound. Both approaches read source from disk (clonefile is transparent to readers) and write `dist/` fresh. ~3 seconds per workspace × 50 = ~150 seconds, the same for both.

If anything, ivk's faster *create* phase (1.8s vs 6.2s at N=10) means ivk's end-to-end time is lower even with identical builds.

## What this means for the LP pitch

### Defensible claims

```text
"ivk creates 100 parallel workspaces of a real project in seconds, using
60× less disk than git worktree — even after each workspace runs a build."
```

This survives. The 60× is the post-clone number; the post-rebuild number narrows to ~8-21× depending on N and project shape, but **both numbers are still dramatic**.

### Claims to NOT make

```text
"Builds are free in ivk."     — FALSE. Builds write fresh artifacts per workspace.
"node_modules are shared."    — Misleading. ivk shares them via clonefile, which
                                breaks the moment something writes to them
                                (rare in practice for node_modules itself).
```

### What an honest LP graph would show

Two lines per scenario (N on X, disk on log Y):
- A solid line for "with builds": shows both A and G have a floor that grows with N.
- A dashed line for "without builds": shows ivk's true flat-ish cost.

This is what [`demos/chart.py`](../demos/chart.py) already produces for the working-tree spike. A follow-up version can add the post-rebuild line.

## Recommended ivk MVP features (per these findings)

1. **`ivk new` already gets the win on disk just from clonefile.** No build-cache feature is required for the basic claim to hold at N=10-50.
2. **Document per-toolchain best-practice config** in `skills/ivk/` so agents know to:
   - **TS / pnpm**: defaults are fine. pnpm already shares its content store.
   - **Rust / Cargo**: default (G) is great if agents don't modify source; for heavier-edit workflows, set `CARGO_TARGET_DIR=$HOME/.cache/ivk/<repo>/cargo-target` so all workspaces share one incremental build cache. ivk should be able to inject this env var on `ivk ws new` when the project root contains a `Cargo.toml`.
   - **Other toolchains**: catalog as the spike expands (Go: `GOCACHE`, Java/Gradle: project caches, Python: `__pycache__` is already cheap).
3. **`ivk ws new --tool` flag** (Phase 1-2): auto-detect or be told which toolchain the project uses, then set the right env vars in the materialized workspace's shell profile. Examples:
   - `ivk new attempt-1 --tool cargo` → injects `CARGO_TARGET_DIR=<shared>`
   - `ivk new attempt-1 --tool turbo` → injects `TURBO_REMOTE_CACHE_HOME=<shared>`
4. **Phase 3+ feature: `ivk cache`** — content-addressed shared cache layer that any tool can opt into. Motivated but not required for v0. The Rust G-shared numbers above suggest a simple "shared dir + env var" approach is good enough for most cases; a full content-addressed store is overkill.

## What still needs to be measured

1. **N=100 for TS** (we ran 10 and 50; 100 expected to extend the linear trend).
2. **Rust at higher N** (we ran 5; 10-20 would be more LP-friendly but each A cell costs ~35s of build time).
3. **Rust with source modifications** between clone and rebuild — validates that cargo's incremental rebuild handles clonefile-shared `target/` correctly.
4. **A larger / more realistic TS fixture** (Next.js with 500+ deps; current fixture is ~64 MB node_modules, real projects are 200-2000 MB).
5. **Real `pnpm install` per worktree on A** — i.e. measure A's *idiomatic* workflow where each worktree runs install separately and pnpm's shared store helps. Our current A bakes node_modules into the worktree, which models the npm/yarn worst case rather than the pnpm sweet spot.

The fixture-baking choice was deliberate: it models the *agent* use case, where each workspace inherits a known-good install rather than spending 5-30 seconds resolving its own. A future spike with "install per worktree" measurement would also be informative.
